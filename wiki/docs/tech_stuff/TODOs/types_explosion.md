## Гипотеза

Для меня ключевым вопросом теперь является: происзодительность системы имеет потенциал, не o(n) от количества типов, а o(log) а в теории и вовсе амортизированные o(1).
Механизм: Схема интексирует типы, все композиты - вектора. Избавление от std::shared_ptr. Memory pool, избыточный reserve по квантам.

## Параллельно с этим

Сужение схем в контексты (не таскать одну гигантскую схему везде, а предрассчитанное множество деревьев (таблица) Scopes. Т.е. для каждого типа есть своя схема зависимых.
Гетерогенные патчи (разные наборы в разных патчах)

## Позиция ИИ

**Сейчас (по сути):** на Writing много путей всё ещё **O(|типов в schema|)** — создать плотный Patch/Future, пройти линии (даже с skip пустых: проверка всё равно по всем слотам, если контейнер плотный). Число *объектов* — другая ось; вчерашний удар был именно по типам.

**DenseTable + индекс** `[0..n)`**:** даёт **O(1) lookup** «дай линию Meta» и дешёвый линейный обход. Это уже не hash. Но **ctor плотного композита** остаётся **O(n типов)**, пока на каждый slot что-то аллоцируешь/инициализируешь. Сам по себе vector не делает commit O(1) от числа типов.

**Когда бывает O(1) / амортизированно по типам:**

- в patch/Future живут **только грязные** линии (или lazy slot): стоимость commit ~ O(|изменённых типов|), часто 1–3 за кадр → практически константа от размера schema;
- schema-index lookup — O(1);
- `O(log n)` от типов почти никому не нужен: типов десятки–сотни, дерево хуже плотного слота; log имел бы смысл у *объектов*, не у aspect’ов.

Итого: гипотеза «не O(n) от количества типов» = верная продуктовая цель; путь к ней — **не плотный обход/создание на каждый Writing**, а dirty/lazy линии + O(1) индекс. DenseTable — ступенька (жёлтый → зеленее lookup), полная гипотеза — когда пустые типы **не участвуют** в горячем пути вообще.

## Кратко по слоям:

1. **Индекс + vector − shared_ptr** — согласен, это must; убирает hash и atomic, не обещает само по себе O(1) commit.
2. **Pool / reserve по квантам** — про ось *объектов* и аллокатор; полезно, но к «взрыву типов» ортогонально. Не смешивать в одной формуле big-O.
3. **Scopes / узкие схемы** — сильный рычаг *сейчас*, без переписывания Patch: hot path (window/camera) не таскает resource::* в Reality. Это pragmatic путь к «жёлтому→зеленее», пока нет lazy линий.
4. **Гетерогенные патчи** — по сути то же, что dirty/lazy: разные наборы линий. Совпадает с тем, что в «Позиция ИИ» названо условием настоящей амортизации.

Про **o(log)**: в тексте гипотезы я бы пометил осторожнее — для *типов* целевой идеал скорее **O(dirty)** ≈ амортизированный O(1) на кадр; log от числа aspect’ов почти не цель. Остальное в блоке ИИ с твоим текстом стыкуется.
