# Reactions scope

В Q1 есть два разных класса реакций.

- `one`-реакции работают на уровне отдельного изменения элемента.
- `all`-реакции работают на уровне всего review-прохода по аспекту.

Именно для `all`-реакций нужна отдельная форма биндинга. Их смысл не в том, чтобы обработать один `id`, а в том, чтобы получить целостную картину текущего proposal: посмотреть весь delta-срез своего аспекта, при необходимости заглянуть в другие аспекты и уже после этого принять решение.

Минимальный пример в Q1:

```q1
one
  !itemRule(-one) // reacts on item deleted
  !itemRule(=one) // reacts on item changed
  !itemRule(>one) // (proposal, not done) reacts on item context changed
  !itemRule(+one) // (proposal, not done) reacts on item added
all
  !localRule(~)
  !globalRule(~BoldEntity)
```

Его рабочая форма в C++:

```cpp
reaction::aspect_wide<MyAttribute>(&MyAttribute::Internals::localRule),
reaction::aspect_wide<MyAttribute, BoldEntity>(&MyAttribute::Internals::globalRule),
```

Смысл этой записи следующий.

## Семантика

Первый параметр шаблона - владелец реакции. Он же всегда входит в множество потенциально слушаемых типов.

Остальные параметры шаблона расширяют это множество. Поэтому:

- `aspect_wide<MyAttribute>` потенциально слушает `MyAttribute`;
- `aspect_wide<MyAttribute, BoldEntity>` потенциально слушает и `MyAttribute`, и `BoldEntity`.

Это соответствует смыслу DSL:

- `!localRule(~)` не указывает тип явно, потому что собственный аспект уже неявно присутствует;
- `!globalRule(~BoldEntity)` не заменяет собственный аспект, а добавляет к нему ещё один источник событий.

Таким образом, `all`-реакция всегда остаётся реакцией аспекта-владельца, даже если фактически анализирует изменения внешнего типа.

## Отличие от `one`

`one`-реакции естественно собираются через специализированные binder'ы вроде:

```cpp
reaction::deletion<MyAttribute>(...);
reaction::constraint::element<MyAttribute>(...);
```

Потому что их форма уже жёстко привязана к одному изменению элемента.

`all`-реакция другой природы. Она вызывается один раз на review и получает `Reacting context`, из которого сама извлекает нужные срезы:

```cpp
context.changes<MyAttribute>()
context.changes<BoldEntity>()
context.proposal.aspect<BoldEntity>()
```

То есть binder `aspect_wide` не должен навязывать item-level семантику. Его задача только одна: объявить владельца реакции и множество типов, изменения которых могут её разбудить.

## Вывод

Форма

```cpp
reaction::aspect_wide<Owner, ExtraSources...>(handler)
```

выражает нужную модель прямо:

- `Owner` - кому принадлежит реакция;
- `Owner` всегда слушается неявно;
- `ExtraSources...` добавляют дополнительные потенциальные источники событий;
- сам handler работает на уровне всего review, а не отдельного элемента.

Для DSL это сохраняет главную идею: `all` - это не «ещё один способ обработать элемент», а отдельный уровень реакции, который мыслит аспектом целиком.

## TODO

Текущие параметры шаблона у `aspect_wide` пока несут в основном декларативный смысл. На сегодняшнем этапе они почти не ограничивают сам `Reacting context`: реакция по-прежнему видит весь proposal и технически может читать и модифицировать всё, что ей доступно через существующий API.

Это сознательный placeholder на будущее. Дальше у `Reactions`-контекста, вероятно, должны появиться как минимум два независимых сужения:

- по типам, которые реакция слушает;
- по типам, на которых реакция вообще имеет право действовать.

То есть шаблонные параметры здесь - не только способ собрать `listens()`, но и заготовка для более строгой модели доступа. Сейчас важнее двигаться дальше по форме DSL и binder'ов, но архитектурный бардак вида «любая реакция видит всё и может сделать всё» рано или поздно придётся прекратить.
